Getting Started With Git

June 14, 2019 7 minutes read

Hi there! Over the past few days I’ve seen people around me struggle with Git. This tutorial is an attempt to get them started in the right direction. This by no means is a complete reference to Git. For a complete reference kindly take a look at Atlassian’s Git Tutorial. I’ve tried to make this content as easy and comprehensible as possible.

What is a VCS?

VCS stands for Version Control System. VCS is a system that records all the modifications done to a file or a project so that developers can recall or refer earlier versions of code (comes in handy if you make a mistake and want to compare earlier versions). There are 2 types of VCSs - Central and Distributed. Central systems have one copy of your project on a server and you commit directly to the central copy. Distributed systems have various copies stored on different machines local to developers rather than one central server.

What is Git?

Git is a type of distributed VCS. It was created by Linus Torvalds in 2005. Developers use Git to keep a track of all the stages that the project goes through to attain it’s current state. It’s one of the most powerful VCS out there with tons of different features. So how does one get started with Git. Well the first step will be to install Git on your system. For this particular tutorial I’ll be using visualize-git. I suggest you do the same and follow along. I find in particularly useful for learning purposes because it removes all the hassle of installing Git on your system and moreover you’ll be able to visualize your whole work flow.

Commited for life!

A commit in a Git repository records a snapshot of all the files in your directory. Whenever you commit changes in a Git repository, Git stores all the changes you’ve made and stores them as a commit. It doesn’t just blindly copy the entire directory but is very smart about keeping track of the set of changes you make. It also maintains a hierarchy of all the commits you make i.e. each commit will have a parent commit. This way you can keep track of the history of changes you’ve made.

Commiting any change in Git is a two part process. First part includes pushing the changes to a staging area. Staging area is analogous to a stopped conveyer belt. You keep all the changes that you’re going to commit on the conveyer belt before moving them forward. Once you’ve added all the changes to the staging area you can commit the changes to move the belt forward.

Staging any change in Git is done by using the command

git add <filename>

PS: Unfortunately visualize-git doesn’t support the above command. I hope that you’ll try the following command on your own in a git repository.

Commiting any staged chages in Git is done using the command (try it on visualize-git)

git commit -m "YOUR COMMIT MESSAGE HERE"

Whenever you commit, I suggest you provide a meaningful message to describe all the changes that have been done in the commit. This will help you when you go back through history of commits. After commiting here is what your workspace on visualize-git should look like -

Branch early, branch often!

Branches in Git are simply pointers to a specific commit. What a branch essentially means is “I want to include the work of this commit and all parent commits”. There is not much storage overhead attached to branches in Git. Hence you can branch as often as you like to keep your work flow clean and organised. PS: Whenever you’re working on an issue or a feature, it is recommended that you create a new branch and work from there. It keeps your work flow clean and avoids future hassles.

In order to create a new branch in Git, type the following command on your visualize-git interface -

git branch newFeature

After you type in the above command you’ll see that a new pointer appears below your last commit.

But right now you’re not checked out on that branch. Any commit you make will move the master branch forward and the not the newFeature branch. In order to select the newFeature branch type the following -

git checkout newFeature

After you checkout, you’ll notice that the HEAD label on the top right changes it’s value to newFeature. This means that anychanges made will be added to newFeature branch now.
PS: HEAD is also a pointer to a commit. It’s the pointer to the commit that you’re currently working on. The above command basically sets the value of HEAD pointer from master to newFeature.

If you try and commit any change now, you’ll see that the newFeature branch moves forward while the master branch still points to the last commit.

git commit -m "New Commit"

Now let’s set the HEAD back to master branch and commit again to see what happens.

git checkout master
git commit -m "New commit to master branch"

As you can see, the two branches start to diverge. This explains the basic concept behind branches. Now each branch can be used to develop a new feature which can later be included in the final branch used for releasing the the code.

Merging and Rebasing

Before proeceeding further I’d like you to set up your workspace by cleaning the above space and creating some commits. I hope you’ll be able to reach this point by the reading material above. We’ll use this sample workspace to understand the concepts of Merging and Rebasing.

We’ll start with Merging. As you’ve seen above branches are really helpful to organise the work flow. But how would one include the commits from one branch into another. The answer is merging. In order to get a visual idea of what I’m talking about checkout to the master branch and type the following -

git merge newFeature

Do you see a new commit? Notice that it has two parents instead of one. Basically this commit includes the changes from both the branches. However you should be aware that if changes in the commits of these branches conflict, Git will throw an error. For example, if last commit from master branch tries to change something and the last commit from newFeature branch tries to change the same code, Git will see it as a conflict and throw an error.

While merging adds a new commit with a parent from each of the branch, rebasing on the other hand takes a set commits and shifts them to a different posistion. Rebasing is used to arrange the commits from different branches in a nice linear fashion. If you feel confused, take a look at the image below.

git checkout sampleBranch
git rebase master

What essentially happened was that all the distinct commits from sampleBranch were copied and based onto the master branch. Just like merging always be aware that conflicting commits will throw an error while rebasing as well.

Moving between commits

In order to move between the commits i.e. to change the value of the HEAD pointer, we use the checkout command. What basically happens when you use the checkout commandis that the HEAD pointer is set to the value you provide. Apart from branch names, value of a pointer can be set relative to a known pointer as well. There are 2 methods for this - the caret (“^”) variable and the tilde (”~“) variable. The caret (”^“) points to the parent of a given pointer. Let’s try it out on a new workspace. Create the below workspace for the upcoming examples -

Now what if one wanted to move the HEAD to one commit behind. Here’s how you do that -

git checkout HEAD^

You can also use caret multiple times -

git checkout HEAD^^^

Caret is good for referring to parent pointers one at a time. But what if you wanted to jump more than one commits in one go. That’s where tilde comes in (”~“). Let’s take a look at an example. In your current workspace, create a new branch and add 5 random commits.

Now let’s checkout HEAD to 3 commits behind master branch.

git checkout master~3

Notice how I used to tilde to refer to 3 commits behind master even though I’m on newBranch.
